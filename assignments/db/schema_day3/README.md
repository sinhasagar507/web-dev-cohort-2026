# Clinic Appointment and Diagnostics Platform ERD

This repository contains the architectural design and database schema for a Refined Clinic Appointment and Diagnostics Platform. The system manages the full patient lifecycle, including scheduling, clinical consultations, financial transactions, and diagnostic reporting.

---

## ERD Notation Guide
The schema utilizes Crow's Foot Notation to define relationships between entities:
* **Single Bar ( | ):** Represents a cardinality of "One."
* **Crow's Foot ( >| ):** Represents a cardinality of "Many."
* **Circle ( O ):** Indicates an "Optional" (Zero) relationship.



---

## Database Associations

### 1. Core Data (1:M)
These relationships define the foundational master data of the hospital system.
* **Specialties to Doctors:** Each Doctor is mapped to a single Specialty (e.g., Pediatrics, Cardiology). A Specialty can have many different Doctors.
* **Doctors to Doctor_Schedules:** Manages availability. A Doctor has multiple Doctor_Schedules entries (defining working day, start time, and end time). Each entry in the schedule belongs to only one Doctor.

### 2. Patient and Appointment Management (1:M)
* **Patients to Appointments:** A Patient can book multiple Appointments over time. Each Appointment is booked by exactly one Patient.
* **Doctors to Appointments:** A Doctor can be scheduled for multiple Appointments. An Appointment is with exactly one Doctor.

### 3. The Consultation Lifecycle (0..1 to 1)
This is the core business logic of the platform, handling the transition from a "booking" to a "medical visit."
* **Appointments to Consultations:** An Appointment (the time slot booking) can optionally become (leads to) exactly one Consultation (the actual medical visit).
    * **Business Logic:** This structure accounts for no-shows or cancellations. If an appointment is not attended, a Consultation record is never created.
* **Walk-In Support:** The app_id in the Consultations table is marked NULLABLE for walk-ins. This means a walk-in consultation can exist without being linked to any appointment.

### 4. Clinical and Financial Linkages (1:M)
* **Doctors/Patients to Consultations:** Established 1:M links where a Doctor conducts multiple Consultations and a Patient makes multiple Consultations.
* **Consultations to Payments:** A single Consultation generates one or more Payments entries. This allows for partial payments, cash + card split, or multiple bills. Each Payment entry is generated by exactly one Consultation.

### 5. Diagnostics and Lab Workflow

#### Many-to-Many (M:N) Resolution
The relationship between Consultations and Lab_Tests is resolved via the Prescribed_Tests junction table.
* **Consultations to Prescribed_Tests:** A doctor uses a consultation to order multiple diagnostic tests.
* **Lab_Tests to Prescribed_Tests:** A standard test definition from the Lab_Tests catalog can be prescribed multiple times (as different orders).
* **Data Integrity:** The Prescribed_Tests entity serves as a bridge. For each prescription, it captures the cost at that specific time to maintain historical pricing accuracy.

#### Strict One-to-One (1:1)
* **Prescribed_Tests to Reports:** Every test ordered must produce exactly one medical report, and every report is for exactly one specific prescription. This is reinforced by a UNIQUE constraint on the report ID.

#### Verification Logic
* **Doctors to Reports:** A doctor verifies multiple reports generated by the diagnostic process. Each report is verified by one specific doctor. The link is marked NULLABLE, meaning a report can exist before a doctor has signed off on it.

---

## Key Features
* **Referential Integrity:** Enforced via Foreign Keys across all modules.
* **Historical Accuracy:** Cost-freezing in diagnostic orders to protect against price fluctuations.
* **Flexible Billing:** Support for multiple transactions per clinical encounter.
* **Audit-Ready Workflow:** Explicit separation between scheduled events and clinical visits.
